Security Considerations

The first security consideration I recommend is regarding personal information that users may be entering into your site. For example, users may be entering credit card details and billing information when purchasing the holidays, or creating user accounts with usernames and passwords so that they can view their holiday purchase history and change details to their holidays. All of these features mean personal information that can be used to identify a person will be stored and pass between the users browser and your server. To protect the user's sensitive information in transit, I recommend implementing HTTPS (Hyper Text Transfer Protocol Secure) to the website. This protocol will encrypt the information between the user's browser and the web server using TLS (Transport Layer Security) preventing men in the middle being able to read the personal information travelling over the network as it will be scrambled and unreadable. HTTPS is easily implemented through purchasing an SSL (Secure Socket Layer) certificate through your web hosting provider (Brafton, 2019). Additionally, once the information has reached the server, it should be fully encrypted inside the database with salted hashes rather than storing the data in plaintext. In the unlikely case an attacker manages to gain unauthorised entry to the database, this will preventing them from helping themselves to a nice list of personal information they could use for fraud or identity theft. Furthermore, when personal information is involved, it is important to be aware of The Data Protection Act 2018 which involves GDPR - the General Data Protection Regulation (Gov, 2021). This regulation states that you have a lawful responsibility to care for user's personal data and keep it secure otherwise severe repercussions may occur. These include potential fines from the ICO (Information Commissioner’s Office) of up to 10 million euros or two percent of the firm's global turnover (whichever is greater), not to mention the reputable damage which will lead to current and future customer's losing trust if their private information is leaked, hence it is important to keep personal information secure (ICO, 2021). Implementing encryption will protect users information, however it is also important to protect users accounts from unauthorised access as this can lead to fraudulent purchases of holidays. An easy way to do this is to add the option for two factor authentication, a simple security mechanism to implement that proves that the user logging into their account is authorised to do so. It does this by requiring a secondary approval method such as a code being sent to their phone to approve the log in attempt. This notifies the user regarding suspicious log in attempts and secures against stolen log in credentials through for example, brute forcing, because an attacker would not be able to log in without the physical secondary device.

Further security considerations relate to security risks from malicious attacking techniques such as SQL injection and XSS. OWASP (The Open Web Application Security Project) provides a list of the top 10 web application security risks and at number one is injection (OWASP, 2021). Most commonly SQL (Structured Query Language) injection, an attacker can take advantage of input fields on your website by entering malicious SQL queries which take advantage of query logic. An attacker may retrieve, modify or delete data from the SQL database without authorisation. Here is an example of an SQL injection that I have previously used in a CTF challenge that will return all data stored in a database: '; SELECT * FROM * WHERE * = ' OR 1=1-- The single quote exits the existing query then introduces a Boolean logic term at the end that will always return true, executing the overall query as true (Messier 2019, p. 426). Another type of injection attack, also in the OWASP top 10 is XSS (Cross-Site Scripting) which allows attackers to inject malicious client-side scripts into an input field that will execute the code within the browser of any user visiting the site (Messier 2019, p. 423). For example, at the simplest level writing "<script>alert(1)</script>" into an input field on a site will produce an alert whenever a user visits that page. Whilst in this example the JavaScript is not malicious in itself, more malicious payloads may be inserted that may harm the user. To prevent these attacks happening, input validation and sanitisation on both the client side and the server side is incredibly important. What I mean by this is only allowing expected payloads to be entered into an input field on the website. For example escaping/sanitising all special characters entered by the user can prevent both SQL injection and XSS. Validation is important on the client side as it helps the user do the right thing before sending their query to the server, but validation is also important on the server side in order to prevent malicious users from skipping the client side form and attacking the server side directly.
